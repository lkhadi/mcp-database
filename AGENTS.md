# Agent Guide for MCP Database Access Server

This file is a reference for AI coding agents working on the **MCP Database Access Server** project.

## Project Overview

This project is a [Model Context Protocol (MCP)](https://modelcontextprotocol.io/) server written in TypeScript that exposes MySQL and PostgreSQL databases to MCP clients such as Claude Desktop and Cursor. It is packaged as the npm package `db-access` for local use; there are no plans to publish it to the npm registry.

Key features:

- Connects to multiple MySQL and PostgreSQL databases simultaneously.
- Exposes exactly **two unified MCP tools** (`list_databases` and `execute_sql`) regardless of how many database connections are configured.
- Optionally connects through SSH tunnels, including multihop jump hosts.
- Supports lazy SSH tunnels that are only established on first use.
- Automatically retries transient database connection errors and recovers SSH-backed connections.
- Blocks dangerous SQL keywords (DROP, TRUNCATE, ALTER, GRANT, REVOKE) by default.

## Technology Stack

- **Runtime**: Node.js >= 18.0.0
- **Language**: TypeScript 5.6+
- **Module system**: ES modules (`"type": "module"`), compiled to NodeNext module resolution
- **MCP SDK**: `@modelcontextprotocol/sdk` ^1.0.0
- **Database drivers**: `mysql2` (MySQL), `pg` (PostgreSQL)
- **SSH**: `ssh2` with `@types/ssh2`
- **Validation**: `zod`
- **Build tool**: `tsc`
- **Dev runner**: `tsx`
- **Test runner**: Node.js built-in `node --test` with `tsx` for TypeScript imports

## Architecture

### Entry Point

`src/index.ts` performs the following steps:

1. Parses `--config` / `-c` CLI argument (defaults to `config.json`).
2. Loads and validates the JSON configuration using `src/config/loader.ts` and `src/config/schema.ts`.
3. Initializes a `ConnectionManager` with all configured database connections.
4. Creates an `McpServer` and registers tools via `registerAllTools`.
5. Connects the server to a `StdioServerTransport` for MCP communication.
6. Handles `SIGINT` and `SIGTERM` by closing all adapters and SSH tunnels.

### Unified Tool Model

The server registers exactly two MCP tools, regardless of the number of configured connections:

- `list_databases` — lists all configured connection IDs and their available databases.
- `execute_sql` — executes SQL against a specified connection and database.

Connection IDs from `config.json` are passed as the `connection` parameter. The tool generator includes the available IDs in the `connection` parameter description.

### Database Adapter Pattern

The `DatabaseAdapter` interface (`src/db/types.ts`) abstracts database-specific operations:

- `execute(sql, database?)`
- `listDatabases()`
- `listTables(database)`
- `describeTable(database, table)`
- `close()`

`MySQLAdapter` uses `mysql2/promise` pools and supports `MAX_EXECUTION_TIME`. `PostgreSQLAdapter` uses `pg.Pool` and supports `statement_timeout`. Both adapters pool connections per database and reset pools after transient connection errors.

### Connection Management

`ConnectionManager` (`src/db/connection-manager.ts`):

- Creates the correct adapter based on `type` (`mysql` or `postgresql`).
- Resolves database lists, auto-discovering with `*`.
- Registers SSH tunnels and supports lazy SSH tunnels (`ssh.lazy: true`).
- Wraps adapters in managed proxies that can recover SSH-backed connections when a transient connection error occurs.

### SSH Tunneling

`SSHTunnelManager` (`src/ssh/tunnel-manager.ts`) builds a chain of SSH connections through optional jump hosts, then creates a local TCP forwarder. It supports:

- Password authentication.
- Private key authentication, with `~` expansion for key paths.
- Multihop jump hosts (`jumpHosts` array).
- Lazy connections (deferred until first use).
- Auto-reconnect on disconnect with exponential backoff (up to 5 attempts).
- Manual recovery triggered by the connection manager on transient SQL errors.

### Connection Retry and Recovery

`src/db/connection-retry.ts` classifies errors as transient by error code (e.g., `ECONNRESET`, `ECONNREFUSED`, PostgreSQL class `08`, MySQL protocol errors) or by message text. The `withConnectionRetry` helper retries once after resetting the affected connection pool. SSH-backed adapters further trigger SSH tunnel reconnection and adapter recreation when a transient error is detected.

### Query Validation

`QueryValidator` (`src/utils/query-validator.ts`) strips SQL comments and string literals, then checks for configured dangerous keywords. When `blockDangerousQueries` is true (the default), queries matching `dangerousPatterns` are rejected before execution.

### Output Formatting

`src/utils/formatter.ts` formats query results as fixed-width text tables, truncating rows to `resultRowLimit` (default 1000). It also provides helpers for column metadata, table lists, and connection lists.

## Configuration

The runtime configuration file is JSON. Use `config.example.json` as the reference; `config.json` is gitignored and intended for local deployment credentials.

### Top-level Schema

```json
{
  "connections": [...],
  "server": { "name": "...", "version": "...", "resultRowLimit": 1000 },
  "security": { "blockDangerousQueries": true, "dangerousPatterns": ["DROP", ...] }
}
```

### Connection Options

| Field | Type | Description |
|-------|------|-------------|
| `id` | string | Lowercase alphanumeric with underscores. Used as the `connection` tool parameter. |
| `type` | `"mysql" \| "postgresql"` | Database type. |
| `host` | string | Database host. |
| `port` | number | Database port. |
| `username` | string | Database username. |
| `password` | string | Database password. |
| `databases` | `string[]` or `"*"` | Databases to expose; `"*"` auto-discovers accessible databases. |
| `ssh` | object (optional) | SSH tunnel configuration. |
| `connectionLimit` | number (default 10) | Connection pool size. |
| `connectTimeout` | number (default 30000) | Connection timeout in milliseconds. |
| `queryTimeout` | number (default 60000) | Per-query timeout in milliseconds. |

### SSH Tunnel Options

| Field | Type | Description |
|-------|------|-------------|
| `enabled` | boolean | Enables SSH tunneling. |
| `lazy` | boolean (default false) | If true, connect on first query. |
| `host` | string | Final SSH hop before the database. |
| `port` | number (default 22) | SSH port. |
| `username` | string | SSH username. |
| `password` | string (optional) | SSH password. |
| `privateKey` | string (optional) | Path to private key file; `~` is expanded. |
| `passphrase` | string (optional) | Passphrase for encrypted keys. |
| `targetHost` | string | Database host from the final SSH server's perspective. |
| `targetPort` | number | Database port on the target side. |
| `jumpHosts` | array (optional) | Ordered list of intermediate SSH jump hosts. |

### Note on `config.json` in the Repository

The repository contains a `config.json` file at the project root with live credentials and extra keys (`cache`, `codeMode`). The current TypeScript source code does **not** recognize those extra keys; validation follows only the schema defined in `src/config/schema.ts`. Treat `config.json` as a local deployment artifact and `config.example.json` as the canonical reference for the schema.

## Code Style Guidelines

- Use TypeScript with `strict: true` and ES2022/NodeNext targets.
- Import paths to other project files must include the `.js` extension, even for TypeScript source files (required by NodeNext module resolution).
- Prefer `async/await` over raw promises.
- Use `console.error` for diagnostic and lifecycle logging; the MCP server communicates over stdio, so `stdout` must remain reserved for MCP messages.
- Keep error messages descriptive and avoid leaking raw stack traces to tool callers.
- Favor explicit interfaces over `any`; shared types live in `src/db/types.ts` and `src/config/schema.ts`.
- Use Zod for runtime configuration validation.
- When adding database-specific behavior, implement it in the appropriate adapter behind the `DatabaseAdapter` interface.

## Testing Instructions

Tests are located in `tests/` and use Node.js's built-in test runner:

```bash
npm test
```

This runs:

```bash
node --test --import tsx tests/**/*.test.ts
```

Current test files:

- `tests/connection-retry.test.ts` — unit tests for transient error classification and `withConnectionRetry`.
- `tests/connection-manager-recovery.test.ts` — integration-style tests for lazy SSH connection recovery through stub tunnel managers and adapters.

Tests rely on stubbed adapters and tunnel managers, so they do not require live databases or SSH servers. When adding new functionality, add corresponding tests in `tests/` using `node:test` and `node:assert/strict`.

## Security Considerations

- **Configuration files contain credentials**: `config.json` stores plaintext passwords and private key paths. It is gitignored, but ensure restrictive file permissions on the deployed config.
- **Dangerous queries are blocked by default**: The server rejects queries matching the configured `dangerousPatterns`. This is a keyword-based guard, not a full SQL parser. Review the pattern list carefully for your environment.
- **Query validation strips literals and comments**: This reduces false positives but may still miss dangerous constructs hidden in unusual syntax.
- **Access control is database-level**: The server relies on the database user's privileges. Do not run with overly privileged accounts.
- **SSH keys**: Use absolute paths or `~`-prefixed paths. Ensure the key files are readable by the process and not committed to version control.
- **No input sanitization for SQL**: The server passes SQL directly to the database driver. Parameterized queries should be used by callers; the server itself does not rewrite or sanitize SQL.
- **MCP stdio transport**: Diagnostic messages must be written to `stderr` only; writing to `stdout` will corrupt the MCP protocol stream.

## Deployment

The package can be run directly after building:

```bash
npm run build
node dist/index.js --config /path/to/config.json
```

For Claude Desktop or Cursor, register the built entry point and a config file in the MCP server configuration:

```json
{
  "mcpServers": {
    "database": {
      "command": "node",
      "args": [
        "/path-to-project/dist/index.js",
        "--config",
        "/path-to-config/config.json"
      ]
    }
  }
}
```

The package defines a `db-access` binary entry point, but it is intended for local use rather than npm publication.
